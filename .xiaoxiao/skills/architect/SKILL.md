# Architect | 架构师

## Layer 1 - 入口

**触发词**: `/architect`, `架构设计`, `设计架构`, `architecture`

**前置条件**: strategy-review 已完成

**核心动作**:
1. 确定系统架构
2. 划分子系统与模块
3. 定义接口与数据流
4. 输出架构文档

---

## Layer 2 - 上下文

### 目标

设计一个可实现、易维护、可扩展的系统架构。

### 完整流程

1. **需求拆解** - 从 `.SPEC.md` 提取技术需求
2. **架构选型** - 选择合适的技术栈和架构模式
3. **子系统划分** - 划分子系统、定义职责
4. **接口设计** - 定义模块间接口和数据流
5. **技术决策** - 关键技术选型及理由
6. **输出架构文档**

### 判断标准

- 架构能支撑当前需求
- 模块边界清晰，职责单一
- 接口设计合理，易于实现
- 技术选型有据可查

### 失败处理

- 需求复杂度过高 → 分层或简化
- 模块耦合过紧 → 重新划分边界
- 技术选型不确定 → 做 POC 验证

### CONFIRM 节点

- 架构选型后：**"我们采用 {架构类型}，原因是 {理由}，确认吗？"**
- 子系统划分后：**"系统划分为 {N} 个子系统：①...②...③...，确认吗？"**
- 架构文档完成后：**"架构设计完成，确认后进入界面设计"**
