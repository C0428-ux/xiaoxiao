# Sprint 规划指南 | Sprint Planning

## 核心目标

将任务合理分配到 sprints，确保可预测的交付。

## Sprint 周期

| 周期 | 优点 | 缺点 |
|------|------|------|
| 1周 | 快速反馈 | 计划开销大 |
| 2周 | 平衡 | - |
| 3周 | 计划少 | 反馈慢 |

## Sprint 容量计算

### 可用人力

```markdown
团队：3人
每人每天：6小时（开会/其他占2小时）
Sprint：2周（10个工作日）

容量 = 3 × 6 × 10 = 180人时
     = 180 / 8 = 22人天
```

### 容量分配

```markdown
总容量：22人天
  - 计划/回顾：2人天
  - 技术难题：2人天
  - 紧急任务：2人天
可用：16人天
```

## Sprint 规划步骤

### 1. 确定 Sprint 目标

```markdown
Sprint 1 目标：
- 用户登录功能上线
- 完成首页UI设计
```

### 2. 选择 PBI

```
优先级从高到低：
☐ P0: 用户登录 ← 选
☐ P0: 首页UI   ← 选
☐ P1: 社交登录   (容量不够，跳过)
```

### 3. 分解任务

```markdown
用户登录（P0）
├── 设计登录UI        4h
├── 设计注册UI        4h
├── 开发登录API       8h
├── 开发注册API       8h
├── 前端登录集成      8h
├── 前端注册集成      8h
├── 测试              8h
└── 部署              4h
────────────────────
总计：52h ≈ 6.5人天
```

### 4. 验证容量

```markdown
已选任务：6.5人天
可用容量：16人天
─────────────
剩余：9.5人天 → 可以再加 P1 任务
```

## Sprint 执行

### 每日站会

```markdown
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻塞？
```

### Sprint 结束

```markdown
1. 演示已完成的功能
2. 回顾做得好的/需要改进的
3. 计算燃尽图
```

## 估算 vs 实际

记录估算和实际的差距：

```markdown
| 任务 | 估算 | 实际 | 差异 |
|------|------|------|------|
| 登录UI | 4h | 5h | +1h |
| 登录API | 8h | 8h | 0 |
```

**目的**：持续校准估算能力
