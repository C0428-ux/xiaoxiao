# 依赖管理指南 | Dependency Management

## 核心目标

识别任务间依赖关系，合理排序，确保高效执行。

## 依赖类型

### 1. 技术依赖

```markdown
任务A（前置） → 任务B（后置）

示例：
数据库设计 → API开发
UI组件开发 → 页面集成
```

### 2. 业务依赖

```markdown
需求确认 → 设计开发
设计评审 → 开发实现
```

### 3. 资源依赖

```markdown
设计师A → 只能做设计A和B
前端资源 → 只能同时做2个任务
```

```markdown
## 依赖关系图

Task A ──┬──> Task D
         │
Task B ──┴──> Task D ──> Task F
         │
Task C ───────────────> Task F
```

## 关键路径识别

### 关键路径 = 完成项目所需的最长时间

```markdown
最长依赖链：
A → B → D → F → G = 15天

其他任务可以并行，但这些必须按顺序：
总工期 = 15天
```

## 依赖处理策略

### 1. 串行化

```markdown
当A是B的前置时：
→ 先完成A
→ 再开始B
```

### 2. 并行化

```markdown
当A和B无依赖时：
→ 同时开始A和B
→ 节省总时间
```

### 3. 解除依赖

```markdown
当A和B互相阻塞时：
→ 寻找共同的中间点
→ 定义清晰的接口
→ 两边可以独立开发
```

## 依赖风险管理

| 风险 | 影响 | 应对 |
|------|------|------|
| 外部依赖延迟 | 阻塞后续任务 | 提前介入、找替代方案 |
| 内部依赖冲突 | 资源争用 | 协调优先级、并行处理 |
| 依赖遗漏 | 后期返工 | 仔细梳理、定期检查 |

## 依赖检查清单

- [ ] 每个任务的前置任务已标注
- [ ] 没有循环依赖（A→B→C→A）
- [ ] 关键路径已识别
- [ ] 外部依赖有负责人
- [ ] 依赖变更时及时更新

## 何时退出

- 所有依赖关系已明确
- 关键路径已识别
- 没有循环依赖
- 依赖风险已评估
