# 流程详解 | tdd-development

## 标准流程

```
Task 取出的顺序：按 P0 → P1 → P2，依赖排序
每个 Task 内：RED → GREEN → REFACTOR
```

---

## Task 开发流程

### Step 1: 取出任务

**做法**:
- 按任务列表顺序取出一个 Task
- 阅读任务描述和预期输出
- 明确完成标准

---

### Step 2: RED - 写测试

**目标**: 写出失败的测试

**做法**:
- 明确"这个功能应该做什么"
- 写出测试用例
- 运行测试，确认失败

---

### Step 3: GREEN - 实现

**目标**: 通过最小代码让测试通过

**做法**:
- 不要写"完美"的代码
- 只写让测试通过所需的最少代码
- 保持简单

---

### Step 4: REFACTOR - 重构

**目标**: 优化代码结构

**做法**:
- 消除重复
- 提升清晰度
- 保持测试通过

---

### Step 5: 验证并继续

**检查**:
- 测试全部通过
- 代码符合规范
- 符合 .SPEC.md 要求

---

## TDD 循环控制

| 指标 | 值 |
|------|---|
| 单任务最大循环 | 10 次 |
| 超时处理 | 移交用户 |
| 循环计数位置 | `skill.loopCount` |

## 测试反模式（来自 Superpowers）

### 铁律

```
1. 永远不要测试 mock 的行为
2. 不要在生产代码中添加仅用于测试的方法
3. 不理解依赖关系就不使用 mock
```

### 反模式 1：测试 Mock 行为

**错误示例**：
```typescript
// ❌ 错误：测试 mock 是否存在
test('renders sidebar', () => {
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**正确做法**：
```typescript
// ✅ 正确：测试真实组件行为
test('renders sidebar', () => {
  render(<Page />);  // 不要 mock sidebar
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});
```

**判断标准**：断言之前问自己——"我在测试真实组件行为还是 mock 的存在？"

### 反模式 2：生产代码中的测试专用方法

**错误示例**：
```typescript
// ❌ destroy() 只在测试中使用
class Session {
  async destroy() { /* 测试专用清理逻辑 */ }
}
```

**正确做法**：测试工具从外部构造，不在生产代码中混入测试 API。

### 反模式 3：无理解依赖的 Mock

**错误示例**：
```typescript
// ❌ 不理解 mock 内部就使用
const mockUserService = { getUser: jest.fn() };
```

**正确做法**：先理解真实依赖的行为，再决定是否需要 mock，以及 mock 什么。

## 验证清单（完成标记）

每个 Task 完成后，验证以下全部通过才能标记为完成：

- [ ] 每个新功能/方法都有对应测试
- [ ] RED 阶段：测试因预期原因失败（功能缺失，不是 typo）
- [ ] GREEN 阶段：最小代码让测试通过
- [ ] REFACTOR 阶段：消除重复、提升命名、不改变行为
- [ ] 测试使用真实代码（除非不可避免才用 mock）
- [ ] 覆盖边界用例和错误情况
- [ ] 输出干净（无错误、无警告）
- [ ] 其他测试全部通过

## Bug Fix 工作流

修复 Bug 时也遵循 TDD：

```
1. 写一个失败的测试，复现 Bug
2. 运行测试确认失败（失败信息应该是预期的错误）
3. 写最小代码修复 Bug
4. 确认测试通过
5. 重构（如需要）
```

**铁律**：永远不要在不写测试的情况下修复 Bug。

## CONFIRM 话术

### 任务开始
> "开始 Task {N}: {名称}，预计 {X} 人天"

### 任务完成
> "Task {N} 完成。RED:{r} / GREEN:{g} / REFACTOR:{f}，继续下一个？"

### 循环超时
> "Task {N} 循环次数达到 {max}，需要人工介入。你想怎么处理？"

### 全部完成
> "开发完成。P0: {N0} 完成，P1: {N1} 完成。确认后进入代码审查。"
