# 范围界定技巧 | Scope Definition

## 核心目标

明确**做什么**和**不做什么**，避免范围蔓延但又不遗漏重要内容。

## 范围定义原则

### 1. MVP 优先

MVP（Minimum Viable Product）= 最少功能满足早期用户验证

```
问题 → 最小解决方案 → 快速验证 → 迭代
```

### 2. 显式排除

不仅要告诉用户"做什么"，还要明确说"不做什么"。

```markdown
### 包括
- 用户注册和登录
- 基本资料管理

### 不包括
- 社交分享功能
- 第三方登录（P2）
- 管理员后台（P2）
```

## 优先级定义

| 优先级 | 定义 | 交付时间 | 说明 |
|--------|------|----------|------|
| P0 | 必须有 | MVP | 没有这个功能产品无法使用 |
| P1 | 应该有 | v1.0 | 重要但不是 MVP |
| P2 | 可以有 | v1.x | 锦上添花 |

## 范围蔓延处理

### 预警信号

- 用户说"顺便加上..."
- "这个很简单，顺便做了"
- 讨论范围逐渐扩大

### 应对话术

```
"这个功能加进来会影响发布时间，我们可以..."
- "先记下来作为 P1"
- "等 MVP 验证后再加"
- "但需要增加 X 天工作量，值吗？"
```

## 范围边界对话模板

```
"MVP 的核心是解决 [核心问题]，我们只需要..."
  ↓
"这些功能会放在 P1：..."
  ↓
"这些明确不在 MVP 范围内：..."
  ↓
"当核心问题被验证后，我们会优先做 P1"
```

## 范围确认检查

- [ ] 核心问题（P0）有明确定义
- [ ] 每个 P0 功能都有用户价值
- [ ] P1/P2 有清晰的边界
- [ ] 用户明确同意范围边界
- [ ] 显式列出了"不包括"的内容

## 何时退出

- P0/MVP 范围明确且用户确认
- P1/P2 有初步定义
- "不包括"的内容已明确说明
- 用户同意优先级排序
