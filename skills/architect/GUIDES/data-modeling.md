# 数据建模指南 | Data Modeling

## 核心目标

设计合理的数据模型，支持业务需求并保持可扩展性。

## 数据库类型选择

### 关系型 (SQL)

**适用场景**：
- 数据结构稳定
- 事务要求高（ACID）
- 复杂查询多
- 关联关系复杂

**代表产品**：PostgreSQL, MySQL

### 文档型 (NoSQL)

**适用场景**：
- 数据结构灵活
- 读多写少
- 不需要复杂关联
- 水平扩展需求

**代表产品**：MongoDB, DynamoDB

### 键值型 (KV)

**适用场景**：
- 缓存
- 会话存储
- 配置
- 简单查询

**代表产品**：Redis, Memcached

### 时序型 (TSDB)

**适用场景**：
- 监控指标
- IoT 数据
- 日志数据

**代表产品**：InfluxDB, TimescaleDB

## 建模步骤

### 1. 识别实体

```
用户、订单、产品、评论...
```

### 2. 确定关系

| 关系 | 示例 |
|------|------|
| 一对一 | 用户 ↔ 用户详情 |
| 一对多 | 用户 → 订单 |
| 多对多 | 用户 ←→ 角色 |

### 3. 定义属性

```markdown
## User
- id: UUID (PK)
- email: String (Unique)
- name: String
- created_at: Timestamp
- updated_at: Timestamp

## Order
- id: UUID (PK)
- user_id: UUID (FK → User)
- status: Enum
- total_amount: Decimal
- created_at: Timestamp
```

## 范式 vs 反范式

### 正常化 (3NF)

```
优点：数据不冗余、更新快
缺点：查询可能需要多表 JOIN
```

### 反正常化

```
优点：查询快、减少 JOIN
缺点：数据冗余、更新复杂
```

**建议**：先正常化，必要时反正常化

## 索引设计

### 何时建索引

- WHERE 条件常用字段
- JOIN 关联字段
- ORDER BY 排序字段

### 索引类型

| 类型 | 适用场景 |
|------|----------|
| B-tree | 范围查询、排序 |
| Hash | 等值查询 |
| GIN | 全文搜索、JSON |
| GiST | 地理数据 |

## 迁移策略

### 原则

1. **向后兼容**：新旧代码同时运行
2. **小步迁移**：一次只改一个小地方
3. **回滚准备**：准备好回滚脚本

### 迁移步骤

```markdown
1. 添加新字段（可空）
2. 部署新代码（写新旧两个字段）
3. 数据迁移（后台跑脚本）
4. 验证数据一致
5. 移除旧字段
```

## 数据保留

提前规划数据保留策略：

```markdown
## 数据保留策略

| 数据类型 | 保留期 | 处理方式 |
|----------|--------|----------|
| 行为日志 | 30天 | 删除 |
| 交易数据 | 永久 | 归档 |
| 用户上传 | 用户删除 | 级联删除 |
| 会话数据 | 7天 | Redis TTL |
```

## 何时退出

- 数据模型已确定
- 关系清晰、无歧义
- 索引已设计
- 迁移策略已规划
