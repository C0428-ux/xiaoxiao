# 估算技术指南 | Estimation Techniques

## 核心目标

将任务分解并估算合理的工作量。

## 估算单位

| 级别 | 小时 | 说明 |
|------|------|------|
| XS | 1-2h | 简单、确定 |
| S | 4h | 标准、可预测 |
| M | 1-2d | 有复杂度 |
| L | 3-5d | 高复杂度 |
| XL | 5+d | 需拆分 |

## 估算方法

### 1. 对比法

```markdown
选择一个已知任务作为基准
↓
对比新任务和基准的复杂度
↓
给出相对估值
```

### 2. 三点估算法

```markdown
乐观估算 (O): 一切顺利
最可能估算 (M): 正常情况
悲观估算 (P): 问题重重

最终估算 = (O + 4M + P) / 6
```

### 3. 故事点法

```markdown
1点 = 1个标准人天

参考任务: 1点
新任务 vs 参考:
- 2倍复杂 → 2点
- 3倍复杂 → 3点
```

## 影响因素

### 增加估算的因素

- 未知技术（需要学习）
- 复杂业务逻辑
- 多方依赖协调
- 质量要求高
- 需要测试覆盖

### 减少估算的因素

- 熟悉的技术栈
- 简单的 CRUD
- 有现成代码参考
- 明确的业务规则

## 估算对话模板

```markdown
"这个任务涉及什么？"
  ↓
"有没有类似的任务可以参考？"
  ↓
"最大的不确定性是什么？"
  ↓
"考虑到这些，估算是多少？"
```

## 缓冲时间

| 任务复杂度 | 建议缓冲 |
|------------|----------|
| 低 | 10% |
| 中 | 20% |
| 高 | 30% |
| 实验性 | 50%+ |

## 常见错误

| 错误 | 问题 | 修正 |
|------|------|------|
| 乐观估算 | 只看顺利情况 | 加缓冲 |
| 忽略研究 | 不确定的技术没算时间 | 分成研究和实现 |
| 忽略测试 | 只估了开发 | 加测试时间 |
| 忽略会议 | 假设纯开发时间 | 加20% |

## 何时退出

- 所有任务都有估算
- 估算经过团队讨论
- 缓冲时间已考虑
- 用户确认估算合理
