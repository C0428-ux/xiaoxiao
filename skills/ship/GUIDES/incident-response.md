# 事件响应指南 | Incident Response

## 核心目标

快速响应和处理生产环境事件，减少影响范围和时间。

## 事件等级

| 等级 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| **P0** | 核心功能不可用 | 即时 | 网站完全无法访问 |
| **P1** | 重要功能受影响 | 15 分钟 | 支付功能不可用 |
| **P2** | 次要功能异常 | 1 小时 | 通知发送延迟 |
| **P3** | 轻微问题 | 4 小时 | UI 显示小问题 |

## 事件响应流程

### 1. 检测与确认

```markdown
问题报告
   ↓
确认问题存在（不是误报）
   ↓
评估影响范围
   ↓
确定事件等级
```

### 2. 通知

```markdown
P0/P1: 立即通知值班 + 相关团队
P2: 30 分钟内通知
P3: 下一个工作日
```

### 3. 调查

```markdown
收集信息：
- 什么时候开始的？
- 有什么变化？（部署、配置、流量）
- 影响多少用户？
- 相关的日志/指标？

定位根因
```

### 4. 止血

```markdown
立即止血（不让问题扩大）
- 回滚到上一版本？
- 关闭问题功能？
- 切换到备用方案？
```

### 5. 修复

```markdown
开发修复
测试验证
部署修复
```

### 6. 恢复

```markdown
确认服务正常
通知用户恢复
持续监控
```

## 事件沟通

### 内部沟通

```markdown
## 事件频道：#incident-YYYY-MM-DD

格式：
[时间] [状态] [更新]

示例：
10:30 发现问题
10:31 确认为 P0
10:32 通知团队
10:35 开始回滚
10:40 回滚完成
10:45 服务恢复
10:50 事件关闭
```

### 外部沟通

```markdown
## 用户通知（如需要）

标题：关于 [问题] 的说明

尊敬的用户：

我们发现 [问题描述]，正在紧急处理中。
预计恢复时间：[时间]

对此造成的不便，我们深表歉意。
```

```markdown

### 改进措施
1. [措施1]
2. [措施2]

### 行动项
| 行动 | 负责人 | 完成日期 |
```

## 何时退出

- 服务恢复正常
- 用户影响已消除
- 根因已定位
- 改进措施已制定
- 复盘已完成
