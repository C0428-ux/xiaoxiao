# 回滚指南 | Rollback Procedures

## 核心目标

在部署失败时快速、安全地回滚到稳定版本。

## 回滚触发条件

| 条件 | 描述 |
|------|------|
| 健康检查失败 | 多次重试仍无法通过 |
| 错误率飙升 | 超过正常基线 3 倍 |
| 关键功能不可用 | 核心功能报错 |
| 性能严重下降 | 响应时间超过 SLA 5 倍 |

## 回滚策略

### 1. 蓝绿部署

```markdown
[Green] 生产环境 ← 当前流量
[Blue]  新版本   ← 备用

问题发生时：
切换流量到 Blue → 停止 Green 的问题版本
```

### 2. 金丝雀发布

```markdown
流量分配：
5%  → 新版本
95% → 稳定版本

发现问题：
逐步减少新版本流量直至完全回滚
```

### 3. 快速回滚

```markdown
当发现问题时：
1. 立即停止新版本
2. 切换到上一稳定版本
3. 保留现场用于排查
```

## 回滚步骤

### Step 1: 确认问题

```markdown
检查指标：
- 错误率是否真的升高？
- 是否是其他原因（网络、依赖）？
- 影响范围有多大？
```

### Step 2: 决策

```markdown
需要回滚吗？
- 问题影响核心功能 → 回滚
- 问题影响少量用户 → 可以继续观察
- 问题可控 → 尝试修复后部署
```

### Step 3: 执行回滚

```bash
# 方法 A: 使用版本标签
git checkout v1.2.3
npm run build
pm2 restart app

# 方法 B: 使用容器
kubectl rollout undo deployment/myapp

# 方法 C: 使用负载均衡
将流量从新版本切回旧版本
```

### Step 4: 验证

```bash
# 健康检查
curl https://api.example.com/health

# 功能验证
运行冒烟测试

# 指标确认
错误率恢复正常
响应时间恢复正常
```

### Step 5: 通知

```markdown
回滚完成后：
1. 通知相关团队
2. 记录事件
3. 安排问题复盘
```

## 回滚时间目标

| 场景 | 目标时间 |
|------|----------|
| 快速回滚 | < 5 分钟 |
| 标准回滚 | < 15 分钟 |
| 复杂回滚 | < 30 分钟 |

## 回滚后处理

### 1. 现场保留

```markdown
不要立即删除问题版本
保留日志、Metrics、Trace
用于问题排查
```

### 2. 事件记录

```markdown
## 回滚事件报告

时间：[时间]
问题：[现象]
原因：[初步判断]
影响：[范围]
回滚：[时长]
改进：[措施]
```

### 3. 复盘会议

```markdown
讨论：
1. 问题根因是什么？
2. 为什么测试没发现？
3. 如何防止再次发生？
```

## 何时退出

- 系统恢复到稳定状态
- 错误率回到正常基线
- 核心功能正常工作
- 用户受影响最小化
